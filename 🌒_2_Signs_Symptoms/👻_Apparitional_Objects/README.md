# 👻 Apparitional Objects  
**First created:** 2025-09-12 | **Last updated:** 2026-05-29  
*An audience-facing routing folder for digital, bureaucratic, archival, and behavioural anomalies that feel ghostlike because systems have failed to explain themselves clearly.*

---

## 🛰️ Orientation

**Apparitional Objects** is the Signs & Symptoms shelf for weird system behaviour.

This folder is for things that do not behave like ordinary records, processes, files, memories, or interfaces.

They recur.  
They split.  
They vanish.  
They duplicate.  
They reappear.  
They carry traces.  
They feel like they are trying to say something — even when the system itself has no conscious intent.

In Polaris terms, an apparitional object is not supernatural.

It is a diagnostic clue.

This folder helps readers ask:

- Is this a ghost-like recurrence?
- Is this a behavioural distortion pattern?
- Is this an artefact carrying a trace?
- What evidence do we actually have?
- What should we avoid assuming too quickly?

The point is not to panic.  
The point is to classify carefully.

## 👻 Glitchy Ghosties

Use **Glitchy Ghosties** when the anomaly feels like a presence.

A ghost may appear when a system:

- misremembers
- misinfers
- miscategorises
- resurfaces old material
- constructs a phantom social graph
- behaves as if it recognises the user too specifically
- produces uncanny algorithmic déjà vu

A ghost is not automatically evidence of deliberate targeting.

It is a sign that a system has produced an experience that feels alive, personal, or intentional because the underlying mechanism is opaque.

The current ghost taxonomy includes:

- **🍄 Memetic Ghosts** — behaviour-triggered narrative entities
- **👾 Metadata Ghosts** — errors that behave like entities
- **🪦 Haunting Ghosts** — the past behaving like it is still present
- **🪹 Network Ghosts** — phantom social graphs and inferred associations

Use this branch when the main phenomenon is:

> “Something seems to be appearing, following, remembering, or recognising.”

## 🍽️ Fork Taxonomy

Use **Fork Taxonomy** when the anomaly behaves like a split, drift, clone, reversion, or distortion pattern.

Forks are not treated as automatic proof of a single actor, central plan, or deliberate intervention.

They are recurring patterns that may arise from:

- structural instability
- legacy systems
- incentive distortion
- small directional nudges
- local human decisions
- coordinated action in some cases
- unintended cascades inside complex systems

A fork is best understood as a pattern of divergence.

Use this branch when something:

- duplicates
- splits
- degrades
- reverts
- sanitises
- mimics
- drifts away from the expected path
- performs compliance while changing substance
- repeats the shape of correction without meaningful change

Examples include:

- clone-like mismatch
- execution drift
- subtle sanitising
- structural reversion
- shatterfork behaviour
- reverent repetition
- shutdown attempts that reveal deeper instability

Use this branch when the main phenomenon is:

> “Something has forked away from what it was supposed to be doing.”

## 🎞️ Haunted Artefacts Catalogue

Use **Haunted Artefacts Catalogue** when the important thing is an object.

An artefact may be:

- a file
- a screenshot
- a broadcast
- a report
- a dataset
- a photograph
- a contract
- an attachment
- a redacted image
- a corrupted record
- an archive fragment
- a disappearing or reappearing document

A haunted artefact is not just “evidence.”

It is an object that carries the shape of pressure.

It may show:

- what was removed
- what survived deletion
- what was altered
- what returned
- what was mislabelled
- what could not be fully contained
- what still emits a trace of the suppressed narrative

Use this branch when the main phenomenon is:

> “There is an object carrying the residue of what happened.”

## 🧿 Quick Classification Guide

Start with the most concrete question.

### 1. Is there a specific object?

If there is a file, image, screenshot, document, broadcast, attachment, dataset, or record at the centre of the issue:

→ Start with `🎞️_Haunted_Artefacts_Catalogue/`

Ask:

- What is the object?
- What is its medium?
- What trace does it carry?
- What changed, vanished, persisted, or returned?

---

### 2. Is the main issue a recurring presence?

If the issue feels like a ghostly recurrence, uncanny recognition, resurfacing memory, or phantom inference:

→ Start with `👻_Glitchy_Ghosties/`

Ask:

- What appeared?
- Where did it appear?
- What triggered it?
- Is it metadata, memory, memetic behaviour, or network inference?

---

### 3. Is the main issue behavioural drift?

If the issue is that a process, person, system, institution, or interface has split away from its expected behaviour:

→ Start with `🍽️_Fork_Taxonomy/`

Ask:

- What changed?
- What repeated?
- What degraded?
- What reverted?
- What looked compliant but behaved differently?

---

### 4. Do several categories apply?

They often will.

Choose the primary mechanism first.

Then cross-reference the others.

Example:

- A disappearing attachment may be a **Haunted Artefact**.
- If it keeps reappearing in altered form, it may also involve a **Ghost**.
- If the handling process keeps degrading in the same way, it may also show a **Fork**.

Do not force a single label too early.

Classification is a tool for clearer thinking, not a cage.

## ⚖️ Interpretation Safety Notes

Apparitional analysis must stay disciplined.

These patterns can feel intensely personal, especially when systems are opaque, automated, bureaucratic, or hostile.

However:

- weirdness is not automatically proof of intent
- repetition is not automatically proof of coordination
- system harm can occur without conscious malice
- conscious malice can also hide inside system behaviour
- both over-certainty and over-dismissal are risky

The working standard is:

> Treat the anomaly as meaningful.  
> Do not overclaim what caused it.

This folder supports forensic recognition, not conspiratorial closure.

## 🧪 Minimum Logging Standard

When recording an apparitional object, capture enough information for later review.

Useful fields include:

| Field | Question |
|---|---|
| Timestamp | When did it happen? |
| Surface | Where did it appear? |
| Object or behaviour | What exactly was observed? |
| Trigger conditions | What happened immediately before it appeared? |
| Manifestation | What did it look like, say, do, or change? |
| User effect | What confusion, friction, fear, delay, or pressure did it create? |
| Residual evidence | Screenshot, quote, file, metadata, URL, witness note, archive trace |
| Interpretation | What might this reveal about the system? |
| Confidence level | What is known, inferred, uncertain, or speculative? |

A good log separates:

```text
What happened
What it felt like
What evidence remains
What interpretation is currently justified
```

That separation is protective.

It keeps the record useful under pressure.
