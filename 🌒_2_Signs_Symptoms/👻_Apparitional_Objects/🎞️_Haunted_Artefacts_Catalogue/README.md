# 🎞️ Haunted Artefacts Catalogue  
**First created:** 2025-10-04 | **Last updated:** 2026-05-30  
*A working catalogue for objects that carry traces of change, disappearance, alteration, persistence, return, or contested interpretation.*

---

## 🛰️ Orientation

**Haunted Artefacts Catalogue** is the apparitional branch for trace-objects.

A haunted artefact is a concrete object that seems to carry the residue of a system event.

It may be:

- a file
- an image
- a report
- a dataset
- a contract
- a screenshot
- a broadcast
- an attachment
- a redacted record
- a corrupted document
- an archive fragment
- a disappearing or reappearing item

The object matters because it gives the weirdness somewhere to land.

It turns “something strange happened” into:

```text
Here is the object.
Here is what changed.
Here is what remains.
Here is what is known.
Here is what is still uncertain.
```

This catalogue does not assume that every altered, missing, corrupted, or resurfacing object has been deliberately suppressed.

Sometimes artefacts carry traces of ordinary system failure: broken links, bad migrations, caching, sync errors, partial deletion, version conflicts, permissions problems, poor record-keeping, or automated processing.

Sometimes they may carry traces of deliberate alteration, concealment, reputational management, or institutional containment.

The purpose of this catalogue is to keep the evidence readable enough that interpretation can happen later.

---

## ⚖️ Personalised Is Not Always Personal

A trace-object can feel personal.

A file may vanish at a meaningful moment.  
An attachment may corrupt in a specific exchange.  
A record may reappear after a dispute.  
A screenshot may preserve something that later becomes inaccessible.  
A document may change in a way that feels pointed.

That does not automatically mean a named person manually targeted the object.

Modern systems can produce object-level weirdness through:

- caching
- sync failure
- permissions errors
- version conflicts
- automated moderation
- data migration
- platform updates
- file conversion
- metadata stripping
- template processing
- link rot
- archive instability
- access-control changes
- third-party integrations
- population-level policy changes

The grounding distinction is:

```text
Personalised does not always mean personally targeted.
Targeted does not always mean individualised.
Individualised does not always mean a human chose you.
```

Even where targeting exists, it may be directed at a category, topic, content type, risk group, archive class, policy area, or public narrative rather than one named individual.

The task is to document the object well enough that later readers can assess what happened without being forced into premature certainty.

The catalogue should preserve:

- the object
- the trace
- the change
- the context
- the possible explanations
- the confidence level

Not just the feeling of being haunted.

Though, honestly, sometimes the feeling is the thing that tells you to look properly.

## ✨ Purpose

This catalogue is for objects that behave strangely in systems of record.

It tracks objects that:

- vanish
- return
- glitch
- duplicate
- degrade
- change form
- lose metadata
- gain metadata
- become inaccessible
- carry redactions
- preserve a deleted moment
- contradict another record
- reveal a mismatch between official story and observable trace

Each entry functions as both evidence and interpretation aid.

The catalogue asks:

> What does this object show, and what can safely be inferred from it?

It does not require certainty at the point of logging.

A fragment is still worth preserving.

A screenshot is still worth naming.

A corrupted file is still a trace.

An inaccessible record may still tell us something about the system around it.

## 🧾 Artefact Entry Structure

Each artefact entry should include the following fields.

| Field | Purpose |
|---|---|
| Object ID | Unique shorthand, e.g. `A-001` |
| Title | Plain-language name for the artefact |
| Medium | File, image, screenshot, dataset, report, broadcast, contract, attachment, etc. |
| Date observed | When the artefact was noticed or captured |
| Source / surface | Where it appeared, disappeared, changed, or was retrieved from |
| Artefact state | Present, missing, altered, corrupted, duplicated, redacted, inaccessible, resurfaced |
| Trace pattern | What repeats, glitches, vanishes, returns, or contradicts another record |
| Possible mechanisms | Technical, administrative, automated, human, institutional, unknown |
| Residual evidence | Screenshot, file hash, quote, metadata, archive link, witness note, version history |
| User effect | Confusion, delay, pressure, evidential loss, friction, reputational effect, emotional impact |
| Interpretation | What the artefact may reveal about the system |
| Confidence level | Known, inferred, uncertain, speculative |
| Cross-references | Related ghost, fork, weirdness screening, field log, or case note |

This structure helps avoid the common failure mode:

```text
The object feels important, but the record does not explain why.
```

The entry should explain why.

## 🧪 Artefact State Guide

Use these labels to describe the object’s current condition.

| State | Meaning |
|---|---|
| Present | The object is currently accessible and stable |
| Missing | The object is expected but cannot be found |
| Inaccessible | The object appears to exist but cannot be opened or reached |
| Altered | The object has changed from an earlier version |
| Corrupted | The object is damaged, unreadable, malformed, or partially broken |
| Duplicated | Multiple versions exist unexpectedly |
| Redacted | Content has been removed, masked, obscured, or withheld |
| Resurfaced | The object reappeared after absence or apparent deletion |
| Contradictory | The object conflicts with another record or official account |
| Unverified | The object exists as a claim, memory, or partial trace but needs confirmation |

Multiple states may apply.

Example:

```text
A document may be both resurfaced and altered.
A screenshot may be present while the original page is missing.
A file may be accessible but stripped of metadata.
```

Do not force the object into one state if the trace is more complicated.

## 🔖 Taxonomy Tags

Use tags for cross-indexing.

Suggested tags:

```text
apparitional, trace-object, metadata residue, redacted image, archive fragment, dataset residue, resurfaced file, missing attachment, corrupted record, version conflict, evidence object, access anomaly, haunting loop, contested record
```

Use the least dramatic accurate tag.

For example:

- `missing attachment` is better than `suppressed evidence` unless suppression is supported.
- `metadata residue` is better than `tampering` unless tampering is supported.
- `access anomaly` is better than `blocked record` unless blocking is supported.

This keeps the catalogue credible.

It also makes stronger claims stronger when the evidence actually supports them.

## 🧭 When To Use This Catalogue

Use **Haunted Artefacts Catalogue** when the central issue is a concrete object.

Start here if the question is:

- What file changed?
- What screenshot remains?
- What record vanished?
- What object resurfaced?
- What document contradicts another document?
- What metadata survived?
- What trace shows the shape of the event?

Use another branch if the object is secondary.

| Main issue | Better destination |
|---|---|
| A ghost-like recurrence or misrecognition | [👻 Glitchy Ghosties](../👻_Glitchy_Ghosties/) |
| A repeated behavioural split or process drift | [🍽️ Fork Taxonomy](../🍽️_Fork_Taxonomy/) |
| An unclear anomaly without enough evidence | [🩻 Weirdness Screening](../../🩻_Weirdness_Screening/) |

If several apply, use this catalogue for the object and cross-reference the other pattern.

## 🧿 Example Classification

```text
A file attached to an email later appears unavailable.
```

Possible artefact entry:

- Primary: 🎞️ Haunted Artefact
- State: missing or inaccessible
- Possible mechanisms: expired link, permission change, platform error, deletion, moderation, deliberate removal, unknown
- Residual evidence: email header, attachment name, screenshot, sender note
- Confidence level: uncertain until more evidence is gathered

```text
The same file later reappears with different metadata.
```

Possible cross-reference:

- 👻 Glitchy Ghosties, if the return creates a haunting pattern
- 🍽️ Fork Taxonomy, if the handling process repeatedly drifts or reverts

The catalogue should preserve the object-level evidence first.

Interpretation comes after.

## ⚖️ Interpretation Safety Notes

Artefacts are seductive.

A changed object can feel like proof.

Sometimes it is.

Sometimes it is proof only that something changed.

Use the following standard:

> Treat the object as meaningful.  
> Do not overclaim what caused the change.

Avoid assuming:

- suppression from disappearance alone
- tampering from metadata loss alone
- coordination from repeated access problems alone
- malice from corruption alone
- innocence from technical explanation alone
- neutrality from official record status alone

A technical fault can still have serious consequences.

A deliberate act can still be disguised as a technical fault.

The catalogue exists so that later analysis has something solid to hold.
