## 🧭 Folder Function

`🍃_Input_Stack` explains what is being fed into the systems described by `🌈_Gold_Pot_Chasers`.

The `🕸️_Core_Problem` folder gives the reader the model:

* there is no pot of gold;
* digital is not side-admin;
* Britain is a damaged house in worsening weather;
* weak seams can become shared exploit surfaces.

This folder asks what loads that model.

The key claim is simple:

> the UK is not dealing with one clean problem.

It is dealing with many overlapping inputs being processed through increasingly shared digital, legal, safeguarding, public-order, procurement, and rights systems.

This folder should stop the reader from flattening the argument into only:

* a cyber problem;
* an AI problem;
* a public-order problem;
* a domestic-abuse problem;
* a hostile-state problem;
* a procurement problem;
* a safeguarding problem;
* a data-governance problem.

The point is that many of those domains now load the same legal, technical, institutional, and cultural seams.

## 🍃 Why “Input Stack”

An input stack is the set of pressures, histories, incentives, reforms, events, and data signals entering a system.

The word “input” matters because bad outputs may not begin at the final decision point.

A bad risk label may come from:

* old records;
* bad categories;
* imported assumptions;
* partner data;
* emergency policy;
* procurement defaults;
* platform signals;
* public-order fear;
* institutional defensiveness;
* a system no one fully owns.

A digital system can only process what it is given.

If the inputs are messy, biased, incomplete, badly governed, or politically loaded, the outputs will not become clean because a dashboard made them prettier.

The input stack is where “garbage in, garbage out” becomes a governance problem.

* diaspora pressure;
* war;
* platform acceleration;
* transnational repression;
* allied-system dependency;
* global vendor ecosystems;

Inputs can include second-order data:

The input stack is not only what a system officially records.

## 📊 Second-Order Data

This folder also defines and routes second-order data carefully.

Second-order data is data around the main event, not necessarily the main event itself.

Examples include:

* timing;
* metadata;
* access failures;
* missing files;
* failed commits;
* recommendation shifts;
* upload failures;
* appeal delays;
* correction-route failures;
* search anomalies;
* visibility changes;
* unexplained account behaviour;
* repeated interface failures;
* mismatched records.

Second-order data is not automatically proof of intent.

It can still be useful.

It helps build pattern logs.

It can show where a system is behaving oddly, failing repeatedly, or producing burdens that official records do not capture.

The discipline is:

* document;
* timestamp;
* preserve;
* compare;
* avoid premature certainty;
* treat artefacts as pattern material, not immediate proof of intent.

## ⚖️ Boundary Discipline

This folder should not turn every input into proof of intent.

It should separate:

* evidence;
* inference;
* hypothesis;
* pattern;
* unknowns;
* audit questions.

It should avoid saying:

> this input proves X actor did Y.

It can say:

* this input creates risk;
* this pressure loads the system;
* this pattern should be audited;
* this category may travel across systems;
* this is where error can become infrastructure;
* this is where old assumptions can become new automation;
* this is where poor data can become coercive output.

The input stack is not the conviction.

It is the map of what the system is being asked to carry.
