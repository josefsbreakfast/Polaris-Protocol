# 🍃 Input Stack  
**First created:** 2026-06-15 | **Last updated:** 2026-06-15  
*Folder orientation for the pressures, histories, reforms, events, and second-order signals that load the Gold Pot Chasers systems map.*

---

## 🧭 Folder Function

- This folder explains the “input stack” behind the `🌈_Gold_Pot_Chasers` cluster.
- It asks what pressures are being fed into Britain’s digital, legal, safeguarding, public-order, procurement, and rights systems.
- It should sit after `1_🕸️_Core_Problem/`.
- The core problem folder explains the model.
- This folder explains what loads the model.
- The key claim:
  - the UK is not dealing with one clean problem;
  - it is dealing with many overlapping inputs being processed through increasingly shared systems.
- This folder should stop the reader from flattening the argument into:
  - “cyber problem”;
  - “AI problem”;
  - “public-order problem”;
  - “domestic-abuse problem”;
  - “hostile-state problem”;
  - “procurement problem.”
- The point is that many of those domains are now loading the same legal, technical, institutional, and cultural seams.

## 🍃 Why “Input Stack”

- An input stack is the set of pressures, histories, incentives, reforms, events, and data signals entering a system.
- The word “input” matters because bad outputs may not begin at the final decision point.
- A bad risk label may come from:
  - old records;
  - bad categories;
  - imported assumptions;
  - partner data;
  - emergency policy;
  - procurement defaults;
  - platform signals;
  - public-order fear;
  - institutional defensiveness;
  - or a system no one fully owns.
- A digital system can only process what it is given.
- If the inputs are messy, biased, incomplete, badly governed, or politically loaded, the outputs will not become clean because a dashboard made them prettier.
- The input stack is where “garbage in, garbage out” becomes a governance problem.

## 🧱 Relationship To The Core Problem

- `1_🕸️_Core_Problem/` establishes:
  - there is no pot of gold;
  - digital is not side-admin;
  - Britain is a damaged house in changing weather;
  - weak seams can become shared exploit surfaces.
- `🍃_Input_Stack/` asks:
  - what is entering the house;
  - what is being fed into the systems;
  - what histories and pressures are being carried forward;
  - what data and categories are doing the load-bearing;
  - where old assumptions become new automation.
- This folder is the bridge between the conceptual model and the later emergency/procurement/actor-pathway analysis.

## 🧺 What Counts As An Input

- Inputs can include formal data:
  - records;
  - risk labels;
  - case notes;
  - flags;
  - referrals;
  - decisions;
  - intelligence summaries;
  - platform signals;
  - procurement requirements;
  - service-access logs.
- Inputs can also include institutional pressure:
  - liability fear;
  - reputational risk;
  - political embarrassment;
  - public-order anxiety;
  - safeguarding panic;
  - secrecy constraints;
  - court backlogs;
  - resource scarcity.
- Inputs can include historical residue:
  - old safeguarding failures;
  - unresolved institutional mistrust;
  - class and regional legitimacy gaps;
  - post-2015 security shifts;
  - austerity damage;
  - analogue records brought into digital systems.
- Inputs can include external pressure:
  - hostile-state activity;
  - diaspora pressure;
  - war;
  - platform acceleration;
  - transnational repression;
  - allied-system dependency;
  - global vendor ecosystems.
- Inputs can include second-order data:
  - timing;
  - access interruptions;
  - metadata;
  - failed uploads;
  - appeal delays;
  - traffic anomalies;
  - recommendation shifts;
  - mismatched records;
  - moderation outcomes.

## 🏛️ Inside-The-House Pressures

- This folder should map domestic pressures internal to the UK system.
- Examples:
  - administrative weakness;
  - poor records;
  - fragmented public bodies;
  - safeguarding failure;
  - domestic-abuse system strain;
  - policing mistrust;
  - justice delays;
  - disability access gaps;
  - austerity aftermath;
  - public-order anxiety;
  - institutional defensiveness;
  - procurement dependency;
  - data-sharing confusion.
- These are “inside the house” because they are not mainly imported from outside.
- They are already present in the system.
- External shocks do more damage when these internal faults remain unrepaired.

## 🌍 Outside-The-House Pressures

- This folder should also map pressures coming from outside or across borders.
- Examples:
  - hostile-state cyber activity;
  - transnational digital repression;
  - platform systems;
  - global vendors;
  - allied intelligence dependency;
  - war;
  - diaspora intimidation;
  - geopolitical pressure;
  - foreign information operations;
  - cross-border organised crime.
- These pressures do not land on a blank system.
- They land on existing records, categories, institutions, contracts, and mistrust.
- The point is not that outside actors create every weakness.
- The point is that outside actors can exploit weaknesses already present.

## 🕰️ Post-2015 Alignment Problem

- The folder should include a node on the post-2015 alignment problem.
- This is not about one single law or event.
- It is about the period where several domains began to intensify or converge:
  - counter-extremism;
  - online safety;
  - coercive-control recognition;
  - platform governance;
  - hostile-state cyber activity;
  - AI and automation;
  - data-sharing;
  - public-order politics;
  - surveillance capitalism;
  - digital identity;
  - procurement modernisation.
- These domains may have been treated as separate policy areas.
- Underneath, they increasingly touch the same systems:
  - records;
  - flags;
  - platforms;
  - portals;
  - case-management systems;
  - risk categories;
  - data-sharing agreements;
  - appeal routes.
- The problem is not simply that 2015 “caused” the stack.
- The problem is that after 2015, many systems began aligning without enough joined-up governance.

## 📊 Second-Order Data

- This folder should define and route second-order data carefully.
- Second-order data is data around the main event, not necessarily the main event itself.
- Examples:
  - timing;
  - metadata;
  - access failures;
  - missing files;
  - failed commits;
  - recommendation shifts;
  - upload failures;
  - appeal delays;
  - correction-route failures;
  - search anomalies;
  - visibility changes.
- Second-order data is not automatically proof of intent.
- It can still be useful.
- It helps build pattern logs.
- It can show where a system is behaving oddly, failing repeatedly, or producing burdens that official records do not capture.

## ⚖️ Boundary Discipline

- This folder should be careful not to turn every input into proof of intent.
- It should separate:
  - evidence;
  - inference;
  - hypothesis;
  - pattern;
  - unknowns;
  - audit questions.
- It should avoid saying:
  - “this input proves X actor did Y.”
- It can say:
  - “this input creates risk”;
  - “this pressure loads the system”;
  - “this pattern should be audited”;
  - “this category may travel across systems”;
  - “this is where error can become infrastructure.”
- The input stack is not the conviction.
- It is the map of what the system is being asked to carry.

## 🧭 Reader Landing

- This folder should leave the reader with one basic insight:
  - bad outputs do not begin at the output stage.
- If the input stack is messy, the system will not become clean because it has been digitised.
- If the categories are wrong, automation can make them travel faster.
- If old records are bad, AI can make old damage look modern.
- If no one owns correction, error becomes infrastructure.
- Before asking what the system decided, ask what the system was fed.
